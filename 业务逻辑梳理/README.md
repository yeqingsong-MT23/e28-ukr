# 业务逻辑梳理

本目录现在是 **UK 报表脚本的提示词源文件目录**，不是普通说明文档，也不是代码注释汇总。

以后新增一个报表时，流程应为：

1. 先新增 `UKXXX.md`
2. 在文档里写清楚生成脚本所需的提示词、业务目标、基础表、粒度、维度、指标、漏斗/阶段、输出 Excel 结构、实现提示
3. 再由 `uk_ai_brief.py` 基于该文档中的提示词生成或辅助生成 `ukxxx.py`

---

## 文档规则

1. **一个 Excel 报表脚本对应一个 Markdown**
   - 例如：`uk101.py` ↔ `UK101.md`
   - 例如：`uk209.py` ↔ `UK209.md`

2. **文档顶部建议有 `UKAI_METADATA`，但不再强制**
   - 推荐保留：这样 `uk_ai_brief.py` 做需求匹配、历史报表检索时会更稳定
   - 如果没有 `UKAI_METADATA`，`uk_ai_brief.py` 也会按文件名和正文自动推断
   - 例如：`UK203.md` 会默认对应 `uk203.py`
   - 基础表族会优先从正文中的 `UK100` / `UK200` 关键字识别，识别不到时再按编码段位推断

3. **正文必须能直接当作代码生成提示词使用**
   - 要明确告诉 `uk_ai_brief.py`：目标脚本名、输出文件、必须落地的 sheet / 字段 / 口径 / Notes / 格式
   - 业务描述不是为了阅读好看，而是为了让模型据此生成 `uk101.py` 这类可直接运行的脚本

4. **正文建议固定包含以下章节**
   - 0. Prompt / 给 `uk_ai_brief.py` 的指令
   - 1. 业务目标
   - 2. 基础文件与数据粒度
   - 3. 核心分析角度
   - 4. 关键维度
   - 5. 核心指标与口径
   - 6. 漏斗 / 阶段 / 状态逻辑（如果有）
   - 7. Excel 输出结构
   - 8. 实现提示

5. **如果当前已经存在一一对应的正式脚本，必须把脚本真实行为反推回 Markdown**
   - 例如：`uk209.py` ↔ `UK209.md`
   - 反推的内容不是泛泛描述，而是要把**真实输出行为**写进 md，尤其是：
     - Sheet 名
     - 列顺序
     - 字段口径
     - Notes 页结构
     - 运行时元信息
     - 排序规则
     - 边界条件与示例
   - 对这种一一对应报表，`md` 不是“参考说明”，而是**日后重新生成 py 的基准文件**
   - 如果发现现有 `py` 和 `md` 不一致，应优先把差异沉淀进 `md`，而不是只口头记住

## `uk_ai_brief.py` 如何使用这些文档

`uk_ai_brief.py` 现在会优先读取本目录中的 Markdown 提示词文档，而不是只依赖代码里写死的报表清单。

它支持两类用途：

1. **需求理解**
   - 根据需求文本匹配最接近的 `UKXXX.md`
   - 提取字段、维度、指标、摘要，生成更贴近业务的分析 prompt

2. **提示词生成代码**
   - 可以直接指定 `--logic-doc UK101`
   - 基于 `UK101.md` 中的提示词生成对应脚本 `uk101.py` 的代码生成 prompt
   - 如配置模型接口，还可以把生成结果直接写成 `.py`
   - 目标是生成像 `uk101.py` 这样可直接落地输出 Excel 的正式脚本
   - `uk_ai_brief.py` 每次执行时都会自动扫描本目录下的 `UK*.md`
   - 如果是**新增文档**，会先按同类型补全提示词：`UK1*` 会参考其他 `UK1*` 文档，`UK2*` 会参考其他 `UK2*` 文档，由 AI 只做“查漏补缺”式补充，不覆盖原主逻辑
   - 新增文档的补充会受固定模板约束，优先补 `Prompt / 业务目标 / 粒度 / 维度 / 指标口径 / 输出结构 / 实现提示 / 代码复刻约束` 这些缺口
   - 自动同步默认只处理**新增 md**、**文档更新时间变化的 md**，以及**对应脚本缺失的 md**
   - 如果是新增或文档更新时间变化，会自动尝试生成并覆盖 `代码\\ukxxx.py`
   - 每次同步后会自动重写 `UK100_start.cmd` 与 `UK200_start.cmd`，把当前已生成的正式脚本按编码顺序自动纳入启动链路
   - 自动同步状态会写入项目根目录的 `业务逻辑梳理同步记录.md`
   - 之前因无模型或失败而留下的旧 `blocked/failed` 文档，不会在后续每次执行时被全量自动重跑；只有再次改动时才会自动重试
   - 后续新增报表时，通常只需要把新提示词文档放进本目录，再执行 `uk_ai_brief.py`；如需显式指定模型，可传 `--model <模型名>`，默认模型为 `gpt-5.4`
   - 若文档名是 `UKXXX.md`，默认目标脚本就是 `代码\\ukxxx.py`

## 判断一份 Markdown 是否写得够好

满足以下条件，才算“可驱动生成代码”的提示词文档：

1. 看完文档能知道：**这张表是为了解决什么业务问题**
2. 能明确：**基础表是什么、粒度是什么**
3. 能回答：**按哪些维度分析**
4. 能回答：**每个指标怎么算**
5. 如果有漏斗，能回答：**每一步怎么定义**
6. 能知道：**最终 Excel 有哪些 sheet、每个 sheet 放什么**
7. `uk_ai_brief.py` 不看旧代码正文，也能基于文档提示词生成对应脚本

## 关于“反推回 md”的要求

如果某张报表已经有成熟正式脚本，例如 `uk103.py`、`uk209.py`，则对应的 `UK103.md`、`UK209.md` 应持续吸收这些正式脚本中已经验证过的实现细节。

建议重点反推这几类内容：

1. **主表和明细表结构**
   - Sheet 名
   - 列顺序
   - 中英文表头

2. **字段与指标口径**
   - 每个字段怎么算
   - 分母/分子定义
   - 去重键
   - 时间窗口
   - 排序和汇总规则

3. **Notes 页要求**
   - 不能只写“有 Notes”
   - 要明确 Notes 里到底写：
     - 字段说明
     - 计算逻辑
     - 示例/业务解释
     - 运行元信息
     - 数据质量统计

4. **实现约束**
   - 核心函数名
   - 公共工具复用方式
   - 必须保留的兼容性行为

这样做的目标是：即使以后正式脚本丢失、备份替换或需要重建，也能仅依据 `md` 重新生成出接近正式脚本的版本。
